# Novita AI Blog Style Analysis

Analysis of proven high-engagement patterns from 200+ published articles on blogs.novita.ai. Use these patterns as the primary reference for title formulas, content structures, and engagement techniques.

---

## I. Title Formulas (by Article Type)

### VS Comparison

**Primary formula:** `[Model A Version] vs [Model B Version]: [Angle/Question]`

Real examples:
- "Qwen 3 Coder vs GLM 4.5: Which Excels in Pure Coding vs Tool-Calling?"
- "DeepSeek V3.1 vs Kimi K2: Which Model Should You Use for Coding？"
- "Llama 4 Maverick vs DeepSeek v3 0324: A Detailed Comparison"
- "GLM 4.5v vs Qwen 2.5 VL: Which Open VLM Should You Use?"
- "GPT-OSS 120B vs Qwen3 235B Thinking 2507: Chat or Code?"
- "Qwen3 Next 80B A3B vs Qwen3 235B A22B 2507: Why Small Excels"
- "GLM 4.5v vs Gemma 3 27B: Should Gemma Users Pay More for GLM?"

**Rules:**
- MUST include exact version numbers on both sides
- Colon introduces the comparison angle or question
- Angle frames a genuine trade-off, not generic "which is better"
- Question format ("Which...?") or declarative angle ("Power vs Versatility")

### VRAM / Hardware

**Primary formula:** `[Model Version] VRAM: [Practical Framing]`

Real examples:
- "MiniMax M2.1 VRAM: 32GB to 500GB Deployment Options"
- "Qwen3 Coder 480B VRAM: How Much Memory Do You Need？"
- "DeepSeek-R1-VRAM-Requirements for Home Servers"
- "Why LLaMA 3.3 70B VRAM Requirements Are a Challenge for Home Servers"
- "GLM 5 VRAM: Cloud vs On-Prem Cost Analysis"
- "Can You Run MiniMax M2.5 Locally? VRAM Reality Check"
- "Qwen3 Next 80B A3B VRAM: Why It Can Rival a 235B Model with Far Less Memory"
- "GLM 4.5v VRAM Setup: Choosing the Right GPU for Multimodal AI"
- "Kimi K2 Thinking VRAM Limits Explained for Cost-Constrained Developers"

**Formulas:**
1. `[Model] VRAM: [Range/Options]` — range communicates completeness
2. `[Model] VRAM: [Question]` — question format for search intent
3. `Why [Model] VRAM Requirements Are a Challenge for [Context]` — pain-point framing
4. `Can You Run [Model] Locally? VRAM Reality Check` — yes/no question hook
5. `How Much VRAM to Run [Model]?` — direct search query match

**Formulas:**
1. `Use [Model] in [Tool]: [Quantified Benefit]` — cost savings / speed / capability

## II. Title Formula Cross-Type Rules

1. **Keyword-first**: Target keyword at the start of the title for SEO
2. **Maximum 10 words** for SEO title variations
3. **Exact versions always**: "DeepSeek V3.2" not "DeepSeek V3", "Llama 3.3 70B" not "Llama 3"
4. **Quantify value**: Include specific numbers when possible (90%, $1/hr, 3 ways, Top 5)
5. **Problem-driven**: Frame as question or challenge ("Why...", "Can you...", "How much...")
6. **Practical context**: Add real-world anchors ("Home Servers", "RTX 4090", "Small Teams")
7. **Colons separate keyword from angle**: `[Keyword]: [Angle]`

## IV. Intro/Hook Patterns (Proven High-Engagement)

### Pattern 1: Pain Point → Thesis (Most Common)
> "Developers building [use case] face a frustrating choice: [trade-off A] or [trade-off B]. **[Model] changes this equation** by delivering [specific advantage] at [specific cost]."

### Pattern 2: Question → Immediate Answer
> "Can you actually run [Model] on consumer hardware? **The short answer: yes, but only with [condition].** Here's what you need to know..."

### Pattern 3: Trend → Why It Matters
> "[Model] is booming — [evidence: downloads, discussions, adoption]. But which API provider gives you the best [cost/speed/reliability]?"

### Pattern 4: Cost Comparison Hook
> "Claude Sonnet 4.5 costs $X/M tokens. [Model] achieves [Y]% of its performance at $Z/M — that's [N]% cheaper. Here's how to set it up."

### Pattern 5: Challenge Framing (VRAM articles)
> "[Model]'s [param count] parameters require [VRAM] at full precision. That's [N] RTX 4090s worth of memory. But quantization changes the math..."

**Rules:**
- First paragraph: state the problem + thesis in bold
- Second paragraph: preview evidence → "Quick Answer" callout box
- NEVER: "In this article, we will explore..."
- Reader should think "yes, that's exactly my situation" in the first sentence

## V. Data Presentation Techniques

### 1. Tables Over Prose (Highest Priority)
Every comparison or spec set uses HTML tables. Never describe specs in paragraphs when a table works.

**Standard table style:**
```html
<table style="width:100%;border-collapse:collapse;margin:20px 0;">
  <thead><tr>
    <th style="background:#7CB342;color:white;padding:12px;text-align:left;">Aspect</th>
    <th style="background:#7CB342;color:white;padding:12px;">Model A</th>
    <th style="background:#7CB342;color:white;padding:12px;">Model B</th>
  </tr></thead>
  <tbody>
    <tr><td style="padding:10px 12px;border-bottom:1px solid #e0e0e0;">...</td>...</tr>
    <tr style="background:#E8F5E9;">...</tr> <!-- alternating highlight -->
  </tbody>
</table>
```

**Rules:**
- Every cell: specific number or "Not disclosed" — never vague text
- Include reference column (e.g., Claude Opus 4.5) for context
- Multiple focused tables > one mega-table
- Green header theme (#7CB342)

## VI. Engagement Techniques

### 1. Thesis-Driven (Not Spec-Listing)
Every article has an opinionated conclusion in the first 2 paragraphs. The thesis IS the article — specs are evidence.

**Per type:**
- **VS:** "[A] wins for [X], [B] wins for [Y]" — not "both are great"
- **VRAM:** "You can/cannot run this on [consumer GPU] — here's what you need"
- **Tool Integration:** "[Model] + [Tool] gives [quantified advantage]"
- **API Provider:** "[Provider] leads on [metric]; choose [Other] for [other metric]"
- **How to Access:** "The fastest path is [method]; choose [other] for [scenario]"

### 2. Quantified Value in Titles AND Body
- "90% Cost Savings" / "$1/Hour" / "80% Cheaper Alternative"
- "32GB to 500GB Deployment Options"
- "Top 3 Providers" / "4 Metrics" / "5 Minutes"
- Numbers create specificity and click motivation

### 3. "Validate Then Enable" (Tool Integration)
Prove the model works (benchmarks + head-to-head) BEFORE teaching setup. Structure:
1. Does [Model] really [capability claim]? ← Validation with data
2. Why [Model] works best with [Tool]? ← Value proposition
3. Which tool should you choose? ← Comparison
4. Setup guide ← Enable

### 4. Scenario-Based Recommendations
Instead of generic "Model A is better," use scenario labels:
- "Terminal-Driven Automation → **Claude Code**"
- "GUI/IDE-based Coding → **Trae**"
- "Open-Source Multi-Provider → **OpenCode**"
- "Cost-Sensitive Environments → **OpenCode or Trae**"

### 5. Community Voices Woven In (Not Separate Section)
Community insights (Reddit, X, HN) integrated into relevant paragraphs:
> "Reddit users on r/LocalLLaMA report 45 tokens/s on dual 4090s with INT4 quantization, confirming our calculated estimates."

NEVER create a standalone "Community Reactions" section. Weave into the relevant technical section.

### 6. Decision Frameworks
End with clear "Choose X if / Choose Y if" tables:
```
| Scenario | Recommendation |
|----------|---------------|
| Terminal automation | Claude Code |
| GUI coding + autocomplete | Trae |
| Multi-provider flexibility | OpenCode |
```

### 7. API as Universal Fallback
Every VRAM and How to Access article includes API access as the final strategy, positioned as "if local deployment is too complex/expensive."

## VII. Anti-Patterns to Avoid

| Anti-Pattern | Example | Fix |
|---|---|---|
| Spec-listing without thesis | "Model has 70B params, 128K context, MoE architecture..." | "70B params with MoE means only 12B active per token — running on a single 4090 with INT4" |
| Generic praise | "amazing performance" | "89% on HumanEval, 12-point improvement over v2" |
| Vague comparisons | "faster than competitors" | "150 tokens/s on A100 vs 100 tokens/s for Llama 70B" |
| Community section standalone | "## Community Reactions" | Weave into relevant technical paragraphs |
| Wrong model versions | "MiniMax M2" instead of "MiniMax M2.1" | EXACT version everywhere, especially FAQ |
| Filler phrases | "In this article we explore..." | Delete. State the fact directly |
| "Both are great" | "Both models have their strengths" | "GLM-5 wins coding (72% SWE-bench), MiniMax wins cost ($0.30/M)" |
| Vague benchmark cells | "Strong performance noted" | "87.3%" or "Not disclosed" |
| Missing source citations | Bare numbers without links | Every number gets inline `<a href>` |
| Decorative symbols | ✨ 🚀 → ★ | No emojis, no decorative Unicode |

## VIII. Conclusion / CTA Structure

### Conclusion (All Types)
- 1 paragraph restating verdict with key numbers
- 1 "Key Takeaway" callout box with actionable recommendation
- No new information introduced
- No filler ("In conclusion", "We hope this helps")

### FAQ (All Types)
- Exactly 5 questions
- Every Q or A mentions model name with EXACT version
- 1-2 sentences per answer MAX
- Practical concerns from real search queries

### SEO Titles (All Types)
- 10 variations
- Keyword-first, problem-oriented
- Max 10 words each
- MUST include specific model version/numbers
- Use formulas from Section I above

## IX. WordPress HTML Rules

- `<h2>` main sections, `<h3>` subsections, `<h4>` sub-subsections
- Code: `<pre><code class="language-python">...</code></pre>`
- All CSS inline (no external stylesheets)
- Links: `target="_blank" rel="noopener"`
- Light green theme: #7CB342 (headers, borders), #E8F5E9 (backgrounds), #CAF6E0 (alternate)
- Tables: green header row, alternating row backgrounds
- Bold for verdicts: `<strong>` tags
- No `<style>` blocks — all inline

---

## X. Source Diversity Requirements

Articles MUST cite diverse source types:
- **Official docs** (HuggingFace model cards, API docs)
- **Blog/review** (at least 2 non-documentation sources)
- **Community** (Reddit, HN, Discord discussions)

NEVER: all sources from documentation only. The mix creates credibility and practical depth.

EXCLUDE: sources about wrong model variants. If a source discusses "DeepSeek V3" but article is about "DeepSeek V3.2", skip it entirely.
